# LEGO Dataset & Classification Reference System — Design Document

## 1. JSON Schema

Each entry in `data/parts.json` follows this schema:

```json
{
  "part_id":              "string  — BrickLink canonical part number (e.g. '3001', '3069b')",
  "name":                 "string  — Exact BrickLink display name",
  "category":             "enum    — Brick | Plate | Tile | Slope | Round | Technic | Special",
  "subcategory":          "string? — e.g. 'SNOT', 'Liftarm', 'Corner' (null if none)",
  "dimensions":           "string  — Human-readable stud footprint, e.g. '2 x 4'",
  "stud_width":           "int     — Width in studs",
  "stud_length":          "int     — Length in studs",
  "height_plates":        "number  — Height in plate units (1 brick = 3, 1 plate = 1)",
  "image_path":           "string  — Relative path to local cached image",
  "image_url":            "string  — Primary Rebrickable CDN URL",
  "embedding":            "float[] | null — 512-dim CLIP vector; null until build script runs",
  "aliases":              "string[]— Alternate part numbers (legacy IDs, lettered suffixes)",
  "tags":                 "string[]— Searchable keywords for pre-filtering before ANN search",
  "bricklink_url":        "string  — Canonical BrickLink catalog page",
  "rebrickable_part_num": "string? — Rebrickable part number when it differs from part_id",
  "year_introduced":      "int?    — First LEGO set year this part appeared",
  "is_obsolete":          "bool    — True for superseded molds"
}
```

**Design decisions:**
- `stud_width`, `stud_length`, `height_plates` are machine-readable for spatial pre-filtering before the embedding ANN search. Filtering by dimension before querying the vector matrix eliminates near-misses between visually similar parts of different sizes.
- `embedding: null` is the initial state. The offline build script populates this and writes the separate `.npy` matrix. For the 35-part seed, embeddings can be stored inline. At scale (>500 parts), they live exclusively in `embeddings.npy`.
- `aliases` drives alias collapse during matching — prevents returning both `3001` and `3001old` as separate candidates.
- `tags` enables a cheap keyword pre-filter that narrows the vector search from 70K → ~500 candidates before any cosine similarity is computed.

## 3. Image Strategy

### Primary Source: Rebrickable CDN

```
https://cdn.rebrickable.com/media/parts/ldraw/0/{part_num}.png
```

**What `/ldraw/0/` means.** This path serves the LDraw isometric render of the part in its default neutral colorway — effectively a color-agnostic silhouette. This is preferred for the reference index because:
- A red 2x4 brick and a blue 2x4 brick produce nearly identical CLIP embeddings when both are sourced from neutral renders, allowing the index to match on shape regardless of the user photo's color
- Every BrickLink part has a `/ldraw/0/` render, even parts never produced in the base color

### Local Caching

The build script downloads every image to `data/images/{part_id}.png` on first run. Subsequent runs skip existing files. The system reads from the local cache at runtime — no CDN dependency after the build.

### Image Preprocessing for CLIP

CLIP ViT-B/32 expects 224x224 RGB input. Rebrickable LDraw renders are ~200x200 with a transparent background. The build script:
1. Opens the PNG, composites on a white background
2. Center-pads to 224x224 (adds white border, does not stretch)
3. Applies CLIP's standard normalization transform

### Priority Fallback Chain

```
1. data/images/{part_id}.png       (local cache — always used at runtime)
2. Rebrickable CDN                 (build script download)
3. BrickLink CDN                   (fallback if Rebrickable has no render)
4. data/images/placeholder.png     (generic LEGO silhouette)
```

## 4. Embedding Strategy

### Model: CLIP ViT-B/32

- **Output:** 512-dimensional float32 vector per image
- **Weights:** `openai/clip-vit-base-patch32` via HuggingFace `transformers` (permissive license)
- **Inference speed:** ~10ms/image on CPU, ~1ms/image on GPU
- **Why ViT-B/32 over ViT-L/14:** ViT-L/14 produces 768-dim embeddings with ~15% better accuracy but is 4× slower and requires more GPU memory. The 512-dim ViT-B/32 is sufficient for shape-matching 70K distinct LEGO geometries.

### Storage Format

| Scale | Embedding Storage |
|-------|-----------------|
| ≤ 500 parts | Inline in `parts.json` as `"embedding": [...]` |
| 500 – 70K parts | Separate `embeddings.npy` + `embeddings_index.json` alignment map |
| > 70K parts | FAISS IVF index (replaces numpy matrix multiply) |

The `embeddings_index.json` maps `part_id → row_index` in the `.npy` matrix:
```json
{"3001": 0, "3002": 1, "3003": 2, ...}
```

A `(70000, 512)` float32 matrix in `.npy` format is 143 MB — loads once at startup, memory-maps efficiently via `numpy.load(mmap_mode='r')`.

### Offline Build Script Design (`scripts/build_embeddings.py`)

1. Load `data/parts.json`, extract `(part_id, image_url, image_path)` tuples
2. For each part: download image if not cached, preprocess to 224x224 white-background RGB
3. Batch through CLIP image encoder (batch size 64)
4. L2-normalize each output vector to unit length
5. Stack into `(N, 512)` float32 array, save as `data/embeddings.npy`
6. Write `data/embeddings_index.json`

Full 35-part seed: < 5 seconds on CPU. Full 70K catalog: ~2 hours on CPU, ~3 minutes on GPU.

## 5. Matching System

### Cosine Similarity

Both query and catalog embeddings are L2-normalized to unit length by the build script. Cosine similarity therefore reduces to a dot product:

```python
# query_vec: (512,) unit vector from CLIP encoding of a detected LEGO crop
# embeddings: (N, 512) unit vectors loaded from embeddings.npy
scores = embeddings @ query_vec  # shape: (N,)
```

This is a single BLAS matrix-vector multiply — under 10ms for N=70K on a modern CPU.

### Threshold Tiers

| Tier | Score | Meaning | UI treatment |
|------|-------|---------|-------------|
| High | ≥ 0.85 | Near-certain shape match | Shown as primary result |
| Medium | 0.70 – 0.85 | Likely match, same family | Flagged "review recommended" |
| Uncertain | < 0.70 | Weak or no correspondence | Top candidates shown for manual review |

Note: LEGO parts are geometrically more similar to each other than natural images, so the noise floor is higher (~0.60) than for general image search. A 1x2 plate and 1x2 tile will score ~0.80–0.88 because their silhouettes differ only in height.

### Top-K Return Format

```json
{
  "top_k": [
    {
      "rank": 1,
      "part_id": "3004",
      "name": "Brick 1 x 2",
      "score": 0.912,
      "tier": "high",
      "category": "Brick"
    },
    {
      "rank": 2,
      "part_id": "3023",
      "name": "Plate 1 x 2",
      "score": 0.847,
      "tier": "medium",
      "category": "Plate"
    }
  ]
}
```

K defaults to 5. The API returns K=10 internally; the UI displays 3 by default with an "expand" option.

### Handling Near-Duplicate Parts

Many parts differ only by mold generation suffix (`3001` vs `3001old`) or are visually identical from a top-down render. Two-pass deduplication:

**Pass 1 — Alias collapse.** Build a `canonical_id` map from the `aliases` field at startup. After scoring, remap any matched `part_id` that appears in an alias list to its canonical ID. Prevents returning `3001` and `3001old` as separate candidates.

**Pass 2 — Geometric deduplication.** After ranking top-K, check if two candidates share the same `category`, `stud_width`, and `stud_length` and their own cosine similarity exceeds 0.92. If so, keep the higher-scoring one only.

**Dimension pre-filter (optional).** When the calling code knows the stud dimensions (e.g., from a YOLO auxiliary head), apply a boolean mask on the embeddings matrix before the dot product:
```python
mask = (parts_df.stud_width == w) & (parts_df.stud_length == l)
scores = embeddings[mask] @ query_vec
```
This narrows the search space by ~95% and eliminates brick/plate/tile confusion for identical footprints.

## 6. Scaling Plan

### Phase 1: Seed (35 parts) — current state
- Single `parts.json` with all metadata
- Embeddings inline or in a tiny `.npy`
- Build script runs in seconds

### Phase 2: Core Catalog (500 – 2,000 parts)
- Introduce `embeddings.npy` + `embeddings_index.json`
- Category-sharded JSON optional: `parts_bricks.json`, `parts_plates.json`, etc.
- Still loaded fully into RAM at startup

### Phase 3: Full Catalog (70,000 parts) via Rebrickable API

**Data source:** Rebrickable provides a free CSV export at `rebrickable.com/downloads/` (no API key required). Fields: `part_num`, `name`, `part_cat_id`. The build script ingests this CSV to populate the catalog.

**Rebrickable API** (free tier, API key required) for per-part metadata:
```
GET https://rebrickable.com/api/v3/lego/parts/{part_num}/
GET https://rebrickable.com/api/v3/lego/part_categories/
```

**Build process at scale:**
1. Download Rebrickable parts CSV → generate `parts.json` with ~70K entries
2. Run build script on GPU instance (Modal one-off job, ~3 min, ~$0.05)
3. Upload `embeddings.npy` to project storage or bundle with app

### Sync Strategy for Monthly BrickLink Releases

1. Re-download Rebrickable CSV every 30 days (it tracks BrickLink additions within days)
2. Diff on `part_id`: new entries queued for image download + embedding
3. Changed `name` or `aliases`: update JSON, no re-embedding needed
4. Store `"last_synced"` timestamp in `parts.json` top-level metadata

### File Format at Scale

| Scale | Format | Notes |
|-------|--------|-------|
| < 2K parts | `parts.json` + `embeddings.npy` | Simple, no deps |
| 2K – 20K | JSON shards by category + `embeddings.npy` | Avoids large JSON parse |
| 20K – 70K | `SQLite` + `embeddings.npy` | Fast filtered lookups before ANN |
| > 70K or multi-user | `Parquet` + `FAISS IVF index` | Sub-ms ANN, columnar projection |

The FAISS transition is the highest-impact scaling step: replaces the O(N) numpy dot product with an O(log N) approximate nearest neighbor search. The build script change is minimal — swap `np.save` for `faiss.write_index`.
